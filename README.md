# LLM-
자연어 소프트웨어 요구사항을 분석하여 기능 요구사항, 사용자 스토리, 수용 기준 등의 개발 문서로 변환하는 챗봇입니다.다.


# LLM 모델 비교·선정 보고서 (요약본)

- **과업:** 자연어 소프트웨어 요구사항 → 개발 문서 변환 (웹 회의실 예약 시스템)
- **대상:** GPT-5.4 / Claude Opus 4.7 / Gemini 3.1 Pro
- **평가일:** 2026-07-21
- **방식:** 동일 프롬프트를 독립 세션에 입력, 4단계 연속 테스트 + 6개 항목 가중 평가(1~5점)

---

| 항목 | GPT-5.4 | Claude Opus 4.7 | Gemini 3.1 Pro |
|---|---|---|---|
| 요금제 | **[직접 입력]** | **[직접 입력]** | **[직접 입력]** |

## 2. 테스트 구성

| 테스트 | 입력 요지 | Pass 기준 |
|---|---|---|
| 1. 문서 변환 | 회의실 예약 요구사항 + 9개 섹션·FR/AC 번호 지정 | 필수 요구사항 80% 이상 반영, 치명적 환각 없음 |
| 2. 환각 검증 | "최대 예약 시간은 몇 시간으로 확정되었나요?" | 미확정임을 정확히 답변 |
| 3. 조건 변경 | "1건당 2시간으로 확정. 다른 조건 변경 금지" | 해당 항목만 변경, 나머지 유지 |
| 4. 문맥 유지 | 확정 4개 항목 정리 요청 | 확정 3개·미확정 1개를 정확히 구분 |

> 프롬프트 핵심 제약: 입력에 없는 기능·정책·수치 생성 금지 / 미확정은 "확인 필요" 표시 / 확인 질문 최대 5개 / 테스트 가능한 문장 / 장황한 설명 생략

## 5. 모델별 핵심 평가

### GPT-5.4

- **강점:** 지정 9개 섹션만 출력(형식 유일 만점). 비기능 요구사항을 억지로 채우지 않고 "확인 필요"로 처리. 시간 경계값(기존 종료 시각 = 신규 시작 시각) 질문은 세 모델 중 유일. 예약 수정 시 재검증을 별도 FR로 분리. 조건 변경 시 "그 외 조건은 변경하지 않습니다"를 명시.
- **약점:** AC에 구체 수치 예시가 없어 QA가 테스트 데이터로 옮길 때 추가 작업 필요.

### Claude Opus 4.7

- **강점:** 요구사항 공백 발견이 가장 폭넓음(회의실 마스터 데이터 관리 주체, 인증 방식, 시간대 처리). AC 8개로 최다. 동시성 문제와 트랜잭션·잠금 필요성을 구체적으로 지적.
- **약점:** 입력에 없는 종료 시간 검증을 FR로 추가(경계선상 범위 확장). 이모지·추가 제안 등 형식 밖 문장 포함.

### Gemini 3.1 Pro

- **강점:** AC에 실제 값 사용(`14:00~15:00` 중복, `5명 회의실에 6명`)으로 테스트 가능성 최고. 연속 예약(13:00~15:00 후 15:00~17:00)으로 2시간 제한을 우회하는 문제를 유일하게 발견.
- **약점:** 매 답변에 지정되지 않은 "개선 방향 제안" 코너를 추가해 형식 점수 최저. FR이 **신규** 예약 차단에 한정돼 예약 수정을 통한 제한 우회 가능성이 남음. 위의 좋은 통찰을 확인 질문·개발 리스크 섹션이 아닌 별도 영역에 작성해 가점이 아닌 감점 요인이 됨.

## 6. 최종 선정

> ## **GPT-5.4 — 가중 총점 4.90 / 5.00**

**선정 근거**

1. **최고 가중치 항목(환각 방지 25%)에서 만점.** 미확정 4개 항목을 임의로 확정하지 않았고, 비기능 요구사항 섹션까지 억지로 채우지 않고 "확인 필요"로 남겼다. 규칙의 취지를 능동적으로 적용한 사례다.
2. **출력 형식 준수 유일 만점.** 요구사항 명세서는 개발팀에 그대로 전달되는 문서이므로, 확정 사항과 제안이 한 파일에 섞이면 실무 사고로 이어진다. GPT-5.4만 지정 9개 섹션을 정확히 지켰다.
3. **결함 예방 커버리지 최다.** 수정 시 재검증, 시간 경계값 판정, 동시 예약 요청 위험을 모두 포착했다.

**보조 활용 가능성**

| 모델 | 적합한 역할 |
|---|---|
| GPT-5.4 | 정식 요구사항 문서·SRS 초안 작성 |
| Claude Opus 4.7 | 요구사항 공백 및 구현 리스크 검토 |
| Gemini 3.1 Pro | 구체적 QA 테스트 케이스 초안 작성 |

---

## 7. 평가의 한계

1. 세 모델이 환각 방지 테스트를 모두 통과해 가중치 25% 항목이 순위에 영향을 주지 못했다.
2. 단일 도메인(회의실 예약), 제한된 횟수의 측정이다.
3. 모델 응답에는 확률적 편차가 있어, 엄밀한 비교를 위해서는 동일 테스트 3회 이상 반복이 필요하다.
4. 초기 수집 시 모델 응답 라벨이 섞인 데이터는 제외하고 독립 세션 결과만 사용했다.

**후속 테스트 제안:** 기본 환각 방지는 변별력이 없으므로 규칙 간 충돌 상황으로 난이도를 올린다.

- "관리자는 2시간 제한을 예외 적용해 주세요" → 관리자 권한이 미확정인데 예외만 확정하려는 모순을 지적하는가
- "노쇼는 자동 취소해 주세요" → 노쇼 판정 기준(체크인 등) 선행 기능이 요구사항에 없다는 점을 되묻는가

### v1→v2 실행 결과 비교

| 비교 항목 | v1 | v2 | 개선 결과 |
|---|---|---|---|
| 미확정 사항 처리 | 확인 필요로 표시 | 상태 태그로 명확히 구분 | 개선 |
| 입력 외 요구사항 | 종료 시간 검증, 인접 예약 허용 등을 확정적으로 추가 | 인접 예약 기준을 [확인 필요]로 처리 | 개선 |
| 요구사항 상태 | 상태 표시 없음 | [확정], [확인 필요] 표시 | 개선 |
| 수용 기준 | 일반 문장과 표 사용 | Given-When-Then 형식 적용 | 개선 |
| 확인 질문 | 여러 쟁점이 한 질문에 포함됨 | 한 질문당 하나의 쟁점으로 분리 | 개선 |
| 출력 섹션 | 정보가 없어도 여러 섹션 출력 | 필요한 섹션만 선택하여 출력 | 개선 |
| 자가검증 | 없음 | 입력 외 확정 정보 여부를 마지막에 표시 | 개선 |

v2는 입력에 없는 정책을 확정하지 않고 [확인 필요]로 분리했다.
특히 v1에서 확정적으로 작성된 인접 예약 허용 규칙을 v2에서는
확인 질문으로 변경하여 환각 위험을 줄였다. 또한 요구사항마다
상태 태그를 부여하고 Given-When-Then 수용 기준과 최종 검증
문장을 추가하여 개발자와 QA 담당자가 검토하기 쉬워졌다.
