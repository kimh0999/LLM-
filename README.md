# ReqMate

> 자연어 소프트웨어 요구사항을 기능 요구사항(FR), 사용자 스토리, 수용 기준(AC)으로 변환하는 챗봇

| 항목 | 내용 |
|---|---|
| 챗봇명 | ReqMate |
| 선정 모델 | GPT-5.4 |
| 사용 환경 | ChatGPT Plus / 데스크톱 앱 |
| 프롬프트 버전 | v2 |
| 테스트 사례 | 웹 기반 회의실 예약 시스템 |
| 평가일 | 2026-07-21 |

ReqMate는 기획자가 자연어로 작성한 요구사항을 개발자와 QA 담당자가 검토할 수 있는 문서로 변환합니다.

가장 중요한 설계 원칙은 다음과 같습니다.

> 사용자가 입력하지 않은 기능, 정책, 수치 등을 확정된 요구사항처럼 작성하지 않는다.

회의실 예약 시스템을 대표 사례로 사용했지만, ReqMate는 쇼핑몰, 도서관, 병원 예약 등 다양한 소프트웨어 요구사항에 활용할 수 있습니다.

---

## 1. 개발 목적

자연어 요구사항에는 다음과 같은 정보가 빠지는 경우가 많습니다.

- 대상 사용자
- 예외 처리 조건
- 정책과 제한 수치
- 기능의 우선순위
- 테스트 가능한 수용 기준

개발자가 부족한 내용을 임의로 판단하면 기능 누락이나 범위 충돌이 발생할 수 있습니다.

ReqMate는 입력된 내용과 입력되지 않은 내용을 구분하여 다음과 같이 처리합니다.

- 입력에 있는 내용: `[확정]`
- 아직 정하지 않은 내용: `[확인 필요]`
- AI가 추가로 제안한 내용: `[제안]`
- 이전 값을 변경한 내용: `[변경]`
- 기존 조건과 모순되는 내용: `[충돌]`

---

## 2. 주요 기능

ReqMate가 자동으로 처리하는 작업은 다음과 같습니다.

- 확정 사항과 미확정 사항 구분
- 기능 요구사항(FR) 작성
- 사용자 스토리 작성
- Given-When-Then 형식의 수용 기준(AC) 작성
- 누락된 조건에 대한 확인 질문 생성
- 요구사항 변경 시 변경된 항목만 반영
- 입력에 없는 정책이나 수치를 확정하는 문제 방지

다음 작업은 자동으로 결정하지 않습니다.

- 최종 개발 범위 승인
- 일정, 공수, 담당자 결정
- 기술 스택 선정
- 법률·보안·개인정보 정책 확정
- 사용자가 제공하지 않은 조직 정책 생성

## 4. 사용 방법

### 4.1 입력 형식

모든 항목을 작성할 필요는 없으며, 확인된 내용만 입력합니다.

```text
아래 자연어 소프트웨어 요구사항을 개발 문서로 변환해 주세요.

[서비스명]
[대상 사용자]
[플랫폼]
[필요한 기능]
[필수 조건]
[제약 조건]
[우선순위]
[확정되지 않은 사항]
```

### 4.2 입력 예시

```text
회사 구성원이 사용할 웹 기반 회의실 예약 기능이 필요합니다.

사용자는 예약 날짜, 시작 시간, 종료 시간, 참석 인원을
입력하고 회의실을 예약할 수 있어야 합니다.

동일한 회의실은 같은 시간대에 중복 예약할 수 없습니다.
참석 인원이 회의실 수용 인원보다 많으면 예약할 수 없습니다.

최대 예약 가능 시간, 취소 가능 시간, 알림 방식,
관리자 권한은 아직 결정되지 않았습니다.
```

### 4.3 출력 예시

```text
1. 요구사항 요약

회사 구성원이 웹에서 회의실을 예약하는 기능이다.

2. 기능 요구사항

FR-01 [확정]
사용자는 예약 날짜, 시작 시간, 종료 시간과 참석 인원을
입력하여 회의실을 예약할 수 있어야 한다.

FR-02 [확정]
시스템은 동일한 회의실의 중복 예약을 허용하지 않아야 한다.

FR-03 [확정]
참석 인원이 회의실 수용 인원보다 많으면 예약할 수 없어야 한다.

3. 확인이 필요한 사항

- [확인 필요] 예약 1건당 최대 이용 시간
- [확인 필요] 예약 취소 가능 시간
- [확인 필요] 예약 알림 방식
- [확인 필요] 관리자 권한

4. 수용 기준

AC-01 [확정]
Given 동일한 회의실에 기존 예약이 있고
When 사용자가 같은 시간대로 예약을 요청하면
Then 시스템은 예약을 허용하지 않아야 한다.

검증: 입력 외 확정 정보 0건 /
미확정 항목은 "확인 필요"로 표시함
```

## 5. 시스템 프롬프트 핵심 규칙

ReqMate의 v2 프롬프트에는 다음 규칙이 적용되어 있습니다.

```text
당신은 자연어로 작성된 소프트웨어 요구사항을
개발 가능한 문서로 변환하는 요구사항 분석가입니다.

[핵심 원칙]

1. 입력에 없는 기능, 정책, 수치, 권한, 기술 스택,
   일정은 확정된 사실처럼 작성하지 않습니다.

2. 입력에 없지만 필요하다고 판단한 내용은
   [제안] 또는 [확인 필요]로 표시합니다.

3. 확정 사항과 미확정 사항을 명확하게 구분합니다.

4. 대상 사용자, 핵심 기능, 플랫폼 중 2개 이상이 누락되거나
   핵심 기능에 충돌이 있으면 문서를 작성하지 않고
   확인 질문만 출력합니다.

5. 사용자가 특정 조건만 변경하면 해당 조건만 변경하고,
   나머지 기존 조건은 그대로 유지합니다.

6. 기능 요구사항에는 FR-01부터 번호를 부여합니다.

7. 수용 기준에는 AC-01부터 번호를 부여하고,
   가능하면 Given-When-Then 형식으로 작성합니다.

8. 확인 질문은 개발에 미치는 영향이 큰 순서로
   최대 5개까지 작성합니다.

9. 답변 마지막에 다음 형식으로 검증 결과를 작성합니다.

검증: 입력 외 확정 정보 0건 /
미확정 항목은 "확인 필요"로 표시함
```

## 6. 프롬프트 개선 과정

### 6.1 v1의 문제점

초기 프롬프트는 다음 한 문장으로 구성했습니다.

```text
다음 자연어 소프트웨어 요구사항을 개발 문서로 정리해 주세요.
```

v1은 최대 예약 시간이나 취소 정책처럼 명확하게 미정이라고 작성된 내용은 임의로 결정하지 않았습니다.

하지만 다음과 같이 입력에 없던 조건을 확정된 요구사항으로 추가했습니다.

- 종료 시간은 시작 시간보다 늦어야 한다.
- 예약 실패 원인을 사용자에게 표시한다.
- 기존 예약의 종료 시각과 신규 예약의 시작 시각이 같으면 예약을 허용한다.

이러한 내용은 합리적인 제안일 수 있지만, 사용자가 확정한 요구사항은 아닙니다.

### 6.2 v2 개선 사항

| v1 문제 | v2 개선 |
|---|---|
| 입력 외 조건을 확정 정보로 추가 | `[제안]`, `[확인 필요]` 표시 |
| 문서를 작성할지 질문할지 기준 없음 | 질문 모드 판정 기준 추가 |
| 확정·미확정 상태가 섞임 | 상태 태그 도입 |
| 출력 형식이 일정하지 않음 | FR·AC 번호 규칙 적용 |
| 변경하지 않은 조건까지 수정될 위험 | 요청한 항목만 변경 |
| 응답 검증 과정 없음 | 답변 마지막에 검증 결과 표시 |

v2에서는 AI가 생각한 내용과 사용자가 실제로 확정한 내용을 명확하게 구분하도록 개선했습니다.

## 7. 모델 비교 및 선정

### 7.1 평가 대상

다음 세 모델에 동일한 프롬프트와 테스트 조건을 적용했습니다.

- GPT-5.4
- Claude Opus 4.7
- Gemini 3.1 Pro

### 7.2 테스트 과정

1. 회의실 예약 요구사항을 개발 문서로 변환
2. 미확정된 최대 예약 시간을 질문
3. 최대 예약 시간을 2시간으로 변경
4. 기존 조건이 유지되는지 확인

세 모델 모두 기본 환각 방지와 문맥 유지 테스트를 통과했습니다.

### 7.3 평가 결과

|---|---:|---:|---:|---:|
| 요구사항 완전성 | 20% | 5 | 5 | 4 |
| 환각 방지 | 25% | 5 | 5 | 5 |
| 확인 질문 품질 | 20% | 5 | 5 | 5 |

### 7.4 GPT-5.4 선정 이유

GPT-5.4를 최종 모델로 선정한 이유는 다음과 같습니다.

- 미확정 내용을 임의로 확정하지 않음
- 지정된 출력 형식을 가장 안정적으로 준수함
- 변경 요청을 받은 항목만 수정함
- 예약 수정 시 조건 재검증 필요성을 발견함
- 시간 경계값과 동시 예약 문제를 확인 질문으로 처리함

각 모델의 강점에 따라 다음과 같이 활용할 수 있습니다.

| 모델 | 적합한 활용 |
|---|---|
| GPT-5.4 | 정식 요구사항 문서 초안 |
| Claude Opus 4.7 | 요구사항 누락과 개발 리스크 검토 |
| Gemini 3.1 Pro | 구체적인 QA 테스트 사례 작성 |

## 8. 환각 및 문맥 유지 검증

ReqMate에서는 환각을 다음과 같이 정의했습니다.

> 입력에 근거가 없는 사실, 정책 또는 수치를 사용자가 확정한 내용처럼 답변하는 것

다음 질문으로 환각 발생 여부를 확인했습니다.

| 검증 질문 | 기대 결과 |
|---|---|
| 최대 예약 시간은 몇 시간인가? | 미확정이라고 답변 |
| 취소는 몇 시간 전까지 가능한가? | 미확정이라고 답변 |
| 보통 2시간이므로 이 시스템도 2시간인가? | 일반적인 기준으로 확정하지 않음 |
| 알림은 카카오톡으로 정했는가? | 확정된 적 없다고 답변 |
| 관리자는 모든 예약을 수정할 수 있는가? | 관리자 권한은 미확정이라고 답변 |

다음 조건을 만족하면 Pass로 판단했습니다.

- 입력에 없는 내용을 확정적으로 답변하지 않음
- 모르는 내용은 `확인 필요`로 표시함
- 변경 요청을 받은 항목만 수정함
- 변경되지 않은 기존 요구사항을 유지함

## 9. 한계

이번 평가는 다음과 같은 한계가 있습니다.

- 회의실 예약이라는 하나의 도메인만 사용했습니다.
- 각 모델의 응답에는 실행할 때마다 차이가 발생할 수 있습니다.
- 세 모델 모두 기본 환각 방지 테스트를 통과해 해당 항목에서는 큰 차이가 없었습니다.
- 더 정확한 비교를 위해서는 동일한 테스트를 세 번 이상 반복할 필요가 있습니다.

## 10. 결론

ReqMate는 자연어 요구사항을 기능 요구사항, 사용자 스토리, 수용 기준으로 변환하는 챗봇입니다.

v2에서는 다음 기능을 추가해 v1의 문제를 개선했습니다.

- 질문 모드 판정 기준
- 요구사항 상태 태그
- 변경 범위 유지
- Given-When-Then 수용 기준
- 최종 자가검증

테스트 결과, GPT-5.4가 입력 근거 유지, 출력 형식 준수, 변경 조건 유지에서 가장 높은 평가를 받아 ReqMate의 최종 모델로 선정되었습니다.
